# 安全生产社会化服务智检系统完成说明

本文档用于说明本项目目前已经完成的建设内容、系统架构、核心业务逻辑、全部功能模块和演示使用方式。它面向项目汇报、答辩展示、后续交接和继续维护。

## 1. 项目定位

本项目是一个面向安全生产社会化服务场景的智能检查系统。系统以企业安全检查为核心入口，围绕企业档案、隐患图片、AI 图像分析、法规知识库、隐患规则、人工复核和正式报告生成形成闭环。

系统当前已经不再只是一个简单的 AI 对话或图片分析页面，而是逐步建设成了一个包含账号权限、后台治理、法规数据、规则判定、报告追溯和工程测试的完整业务系统。

当前系统的核心价值是：

- 用企业档案组织安全检查数据。
- 用本地法规知识库保存公开法律法规、国家标准和行业标准条文。
- 用规则库把法规条文转化为可以执行的隐患判定规则。
- 用 AI 做图片场景识别和可见事实抽取，而不是让 AI 凭空决定隐患等级。
- 用后端规则、证据和人工复核共同决定最终报告内容。
- 用 Word/PDF 报告输出可交付成果。
- 用 GitHub PR、CI、测试和文档保证每一步修改可追溯。

## 2. 你完成了什么

### 2.1 从原型到稳定工程主线

项目最初主要是普通用户上传图片、调用 AI、生成报告的演示链路。后续经过多轮重构后，已经完成以下工程化建设：

- 建立新的 GitHub 仓库主线，并以稳定版本作为 `main` 基线。
- 建立按 PR 分阶段推进的协作方式。
- 增加 PR 描述模板，要求每次变更说明问题、实现方式、验证记录和风险检查。
- 建立 CI 流程，支持后端检查、后端测试、前端 H5 构建和 Docker 构建。
- 明确本地开发同步规范，避免每次合并后重复配置 `.env`、数据库和 AI Key。
- 把开发中的临时资料、原始文件、密钥和本地数据库配置排除在 Git 提交之外。

### 2.2 完成认证与权限收口

系统已经从早期信任 `user_id`、`admin_id`、`X-Admin-Id` 的方式，升级为后端统一鉴权。

已经完成：

- 登录返回 Bearer Token。
- 后端使用 JWT Access Token 识别当前用户。
- 使用 `auth_sessions` 表保存登录会话，用于撤销、过期、审计和强制失效。
- 增加 `GET /api/auth/me` 获取当前登录用户。
- 增加 `POST /api/logout` 退出登录。
- 普通用户接口不再信任前端传入的 `user_id`。
- 管理员接口不再信任伪造的 `admin_id` 或请求头。
- 普通用户、管理员、权限点均由后端统一判断。

核心权限包括：

- `enterprise:manage`：企业信息维护。

### 2.8 完成 PDF/DOCX 草稿审核流程

考虑到 PDF、Word 文档解析可能出现乱码、错位、条款识别错误等问题，系统没有让抽取结果直接进入正式知识库，而是加入了草稿审核机制。

已经完成：

- 新增 `knowledge_clause_drafts` 草稿表。
- 管理员上传 PDF/DOCX 后，系统尝试抽取条文。
- 抽取结果先进入草稿池。
- 管理员可以编辑草稿、通过草稿或驳回草稿。
- 审核通过后才写入正式法规条文库。
- 驳回后不会进入正式库。
- DOC 旧格式只保留文件级管理，不自动抽取。
- OCR 暂不完整实现，只保留字段和流程预留。

这保证了系统不会把脏数据、乱码条文、错误条款直接作为 AI 判断依据。

### 2.9 完成隐患规则库

系统已经建立了“法规条文 -> 隐患规则 -> AI 分析结果”的中间层。

已经完成：

- 新增隐患规则表。
- 新增规则与法规条文关联表。
- 管理员可以新增、编辑、启用、停用、归档规则。
- 管理员可以搜索并关联已校验法规条文。

- 规则只能关联 `verified` 且现行有效的正式条文。
- 没有依据的重大隐患规则不能启用。
- 提供第一批高频隐患规则种子。

第一批规则覆盖方向包括：

- 消防通道堵塞。
- 安全出口堵塞。
- 消防设施器材被遮挡或挪用。

- 高处作业分级与防护。
- 有限空间作业风险。
- 特种设备重大事故隐患判定。

### 2.10 完成 AI 场景门控与规则驱动判断

AI 分析逻辑已经从“直接让大模型给结论”升级为分层判断流程。

当前流程是：

1. AI 先判断图片是否属于安全生产检查场景。
2. 如果图片无关，例如动漫图、风景图、生活照，系统拒绝生成正式报告。
3. 如果图片属于现场检查场景，AI 只抽取可见事实。
4. 后端根据可见事实、用户描述、企业行业和规则库检索候选规则。
5. AI 只能在候选规则中选择匹配项。
6. 后端校验 AI 返回的规则 ID 是否真实存在。
7. 隐患等级由规则、证据充分性和人工复核状态共同决定。
8. 重大隐患必须有本地规则和法规条文支撑。

输出字段包括：

- `scene_status`：场景状态。
- `scene_reason`：场景判断理由。
- `visible_facts`：可见事实。
- `uncertain_points`：不确定点。
- `matched_rules`：命中规则。
- `legal_refs`：法规依据。
- `evidence_sufficiency`：证据充分性。
- `hazard_level`：隐患等级。
- `review_required`：是否需要人工复核。

当前支持的业务结论包括：

- 非业务图片。
- 未发现明显隐患。
- 疑似隐患。
- 一般隐患。
- 疑似重大隐患。
- 重大隐患。
- 需人工复核。

报告 V2 内容包括：

## 6. 核心业务流程

### 6.1 用户分析流程

```text
用户登录
  -> 维护企业信息
  -> 上传隐患图片
  -> 选择 AI 模型
  -> 输入检查要求
  -> 发起分析
  -> AI 判断是否为安全生产检查场景
  -> AI 抽取图片可见事实
  -> 后端检索规则库
  -> AI 在候选规则中选择匹配项
  -> 后端校验规则和法规依据
  -> 生成 AI 初判结果
  -> 用户或管理员人工确认
  -> 生成正式 Word/PDF 报告
  -> 历史记录归档
```

### 6.2 法规条文入库流程

```text
收集官方法规或标准
  -> 整理为 CSV
  -> 执行质量校验
  -> 通过后导入 knowledge_clauses
  -> 标记来源、分类、文号、条款号和校验状态
  -> 覆盖率看板展示
  -> 可被规则库引用
```

### 6.3 PDF/DOCX 文档入库流程

```text
管理员上传 PDF/DOCX
  -> 系统尝试抽取条款
  -> 写入 knowledge_clause_drafts 草稿池
  -> 管理员人工编辑和审核
  -> 审核通过后写入正式条文库
  -> 驳回则不进入正式库
```

### 6.4 隐患规则维护流程

```text
管理员创建规则
  -> 填写规则名称、分类、等级、关键词、触发条件、证据要求
  -> 关联已校验法规条文
  -> 后端校验规则合法性
  -> 启用规则
  -> AI 分析时作为判定依据
```

### 6.5 报告生成流程

```text
AI 初判完成
  -> 保存分析记录
  -> 保存命中规则和法规条文引用
  -> 判断是否允许生成正式报告
  -> 需要人工确认时等待确认
  -> 确认后生成 Word
  -> 转换或生成 PDF
  -> 保存报告路径
  -> 用户下载
```

## 8. 当前系统的核心竞争力

项目当前最重要的升级点，是把“AI 生成一段看起来像报告的文本”，改造成了“法规知识库 + 规则库 + AI 事实抽取 + 人工复核 + 可追溯报告”的组合。

具体体现为：

- AI 不再直接拥有最终定性权。
- 重大隐患必须有规则和法规条文支撑。
- 无关图片不会被硬套成一般隐患。
- 知识库为空或规则不足时，系统会进入需人工复核，而不是编造法规依据。
- 报告中可以追溯到命中的规则和法规条文。
- 正式报告生成前增加人工确认。
- 法规条文作为本地数据资产，可持续扩充。
- 后续使用者可以继续导入新的法规、标准和规则，让系统覆盖更多行业场景。

## 12. 当前边界与说明

虽然当前系统已经形成完整闭环，但仍需要明确以下边界：

- AI 视觉识别仍可能出现不稳定，系统通过规则和人工复核降低风险。
- PDF/DOCX 自动抽取不能完全保证条文准确，所以必须经过草稿审核。
- OCR 没有完整实现，扫描件和拍照件不建议作为自动抽取来源。
- 重大隐患判断依赖规则库和条文质量，规则不足时应进入人工复核。
- 自定义 Word 模板需要满足占位符要求，否则报告内容可能不完整。
- 高质量 PDF 转换在 Windows + Microsoft Word 环境下效果最好。
- 真实 AI 调用依赖管理员配置的模型服务、API Key、模型能力和网络环境。
- 公共 CI 不使用真实数据库密码和真实 API Key。
