# 后续阶段开发计划

## 阶段原则

- 每个阶段只聚焦一类目标
- 每个阶段都要形成可检查的交付物
- 每次合并必须附带验证记录

## 阶段 0：新仓库接管

目标：

- 用 `merge-stable-report` 建立新仓库主线
- 启用 CI、PR 模板、Issue 模板
- 统一团队分支策略与本地配置流程

交付物：

- 新仓库 `main`
- `.github/workflows/ci.yml`
- `NEW_REPO_WORKFLOW.md`
- `project/DEPLOY_DOCKER.md`

## 阶段 1：配置与安全加固

目标：

- 完成 AI 模型配置安全整改
- 明确敏感信息管理边界
- 避免真实密钥进入仓库、日志和接口返回

交付物：

- 接口文档更新
- 测试记录更新
- 配置模板与使用说明更新

## 阶段 2：管理员端后台真实联调

目标：

- 知识库
- AI 模型配置
- 日志
- 报告模板
- 数据备份
- 工作台统计

交付物：

- 每个模块独立 PR
- 联调记录
- 必要的数据库和接口文档更新

## 阶段 3：普通用户端优化

目标：

- 继续打磨 `process.vue`
- 检查权限收口
- 复核企业归属与数据隔离
- 做 H5 / 微信双端回归

交付物：

- 功能测试记录
- 界面说明更新
- 风险项清单

## 阶段 4：质量体系完善

目标：

- 增加更稳定的自动检查
- 整理回归清单
- 形成里程碑发布节奏

交付物：

- 最小回归测试规范
- 发布说明模板
- 阶段验收清单
