# 方案 B 分阶段推进路线

本文档用于约束新仓库后续按方案 B 递进式改造，保证每一步都可分支、可审查、可回滚、可复核。

## 1. 推进原则

所有阶段都遵循以下规则：

1. 每个阶段拆成一个或多个小 PR
2. 每个 PR 只处理一个明确目标
3. 不把文档整理、配置整改、业务改造、数据库迁移混成一个大提交
4. 每个阶段同步更新：
   - `project/API_DOC.md`
   - `project/DESIGN_DOC.md`
   - `project/TEST_RECORD.md`
5. 如涉及数据库结构，必须同步更新 `project/database/schema.sql`

## 2. Phase 0：仓库治理基线

目标：

- 让新仓库从第一步开始具备稳定的 GitHub 协作边界
- 修正文档中的旧分支描述
- 规范环境变量模板，避免默认敏感值继续传播

建议 PR：

- `chore: establish repository config baseline`

交付物：

- 更新根目录 README
- 更新 `NEW_REPO_WORKFLOW.md`
- 更新 `.env.docker.example`
- 更新 `project/backend/.env.example`
- 更新 Docker 部署说明

验收标准：

- CI 正常触发
- PR 模板正常承接
- 团队成员可按文档独立完成本地配置

## 3. Phase 1：用户态认证与核心接口收口

目标：

- 建立统一登录态
- 普通用户核心接口不再信任前端传入的 `user_id`
- 修复横向越权风险

重点范围：

- 登录接口返回可校验的登录态
- 新增用户态鉴权中间件
- 收口以下核心接口：
  - 企业信息读取/更新
  - 隐患图片上传/删除/标注
  - 智能分析
  - 历史记录读取/修改/删除
  - 会话读取/删除

建议拆分：

1. `feature/auth-user-session-foundation`
2. `fix/auth-user-route-hardening`

验收标准：

- 伪造其他用户 ID 无法读写他人数据
- 普通用户主链路保持可用
- H5 与小程序端登录后流程可回归

## 4. Phase 2：管理员态鉴权与 AI 配置安全

目标：

- 管理员接口改为真正后端鉴权
- AI 模型配置具备最小安全治理能力

重点范围：

- 管理员登录态与管理员中间件重构
- `admin_id` 不再作为可信身份来源
- AI 模型配置：
  - API Key 加密存储
  - 列表脱敏显示
  - 更新字段白名单
  - 激活切换事务化
  - 禁止删除当前启用模型

建议拆分：

1. `feature/auth-admin-session-hardening`
2. `fix/model-config-security-baseline`

验收标准：

- 管理员接口不能靠伪造 `X-Admin-Id` 访问
- 模型配置页开始使用真实安全接口
- 配置修改过程可追踪、可回滚

## 5. Phase 3：后台治理模块真实联调

目标：

- 把当前仍使用 mock 数据的后台模块逐步接入真实接口
- 提升管理端的可交付性和可信度

优先顺序：

1. 模型配置
2. 操作日志
3. 工作台统计
4. 报告模板
5. 数据备份
6. 知识库管理

建议原则：

- 每个模块单独一个 PR
- 先接真实读取，再接增删改
- 前端联调前先把后端接口、权限、日志补齐

验收标准：

- 页面展示数据来自真实后端
- 高风险操作具备明确提示和后端保护
- 关键管理操作可记录到日志

## 6. Phase 4：知识依据追溯与产品闭环增强

目标：

- 让知识库和报告依据更像专业系统，而不是简单文件台账

重点范围：

- 法规/标准条款化
- 分类、版本、来源信息
- 报告中引用依据可追溯
- 企业端和管理员端可复核分析依据

建议拆分：

1. `feature/knowledge-clause-structure`
2. `feature/report-reference-traceability`

验收标准：

- 知识依据在界面和报告中可定位来源
- 关键引用信息可以人工复核

## 7. Phase 5：测试与发布治理

目标：

- 让项目进入可持续演进状态，而不是只能依赖人工试跑

重点范围：

- 增加最小集成测试
- CI 分层
- Docker 启动检查
- 发布说明模板
- 回归清单

建议拆分：

1. `test/minimal-api-integration`
2. `chore/ci-layering`
3. `docs/release-checklist`

验收标准：

- 至少覆盖登录、鉴权、主链路、模型配置、报告生成几个关键点
- 每次 PR 都有明确验证记录

## 8. 每阶段输出模板

每个阶段合并前至少补齐以下内容：

- 变更目标
- 影响范围
- 接口变更
- 数据库变更
- 配置变更
- 本地验证命令
- 手动验证步骤
- 已知风险

推荐在 PR 中明确写出：

- 本阶段解决了什么
- 本阶段没有解决什么
- 下一阶段接什么
